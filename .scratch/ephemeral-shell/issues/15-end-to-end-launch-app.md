# 15 — End-to-end launch app

**What to build:** The flake's default app — an interactive fzf preset picker that, once a preset is chosen, wires together isolation (ticket 12) to enter an ephemeral `$HOME`, secrets bootstrap (ticket 14) to decrypt secrets into it, the core module's file-write plan (ticket 11) to materialize static/fragment files into it, and package assembly (ticket 13) to drop into `nix shell` with the resolved package set. This is the first ticket that's demoable as a full launch, even against a minimal test preset/bundle/fragment fixture rather than real content.

**Blocked by:** 11 — Core materialize module, 12 — Isolation launcher, 13 — Flake package assembly, 14 — Secrets bootstrap

**Status:** done

- [x] `nix run` on this flake presents an interactive fzf picker listing available presets
- [x] Selecting a preset enters an isolated ephemeral `$HOME` (ticket 12)
- [x] Secrets are decrypted and placed into that `$HOME` (ticket 14)
- [x] Static and composed-fragment files are materialized into that `$HOME` per the core module's file-write plan (ticket 11)
- [x] The session drops into `nix shell` with the preset's resolved package set (ticket 13)
- [x] Demoable end-to-end using at least a minimal test preset/bundle/fragment fixture
- [x] On exit (normal or Ctrl-C), nothing from the session survives outside the wiped ephemeral `$HOME`

## Comments

Implemented as `bin/launch` (fzf picker + isolate hand-off), `bin/launch-inner` (the wiring itself), and `flake.nix`'s new `apps.<system>.default` (verified with `bin/launch-demo`).

- **`bin/launch`** resolves `ROOT` from its own path (same `dirname "$0"/..` pattern as `bin/assemble`), lists `presets/*.json`, runs `fzf` over the names, then `exec`s `bin/isolate -- bin/launch-inner "$ROOT" "$PRESET"`. No new selection logic — `fzf` picks, `isolate` isolates, nothing in between.
- **`bin/launch-inner`** is the new glue, Python (not bash) since its job — call ticket 14's `secrets.bootstrap`, then ticket 11's `materialize.build_plan`, then resolve ticket 11's `load_preset(...).bundles` — is inherently calling into the existing Python core, same reasoning ticket 14 gave for staying Python. Its public `build_home(root, preset_name, target)` does the decrypt + write + resolve and returns the `bin/assemble` argv rather than exec'ing directly, so it's callable/testable in-process; `__main__` validates `ROOT`/`PRESET` were given (usage message + exit 1 otherwise, matching `bin/assemble`'s own bad-input handling) then does the `os.execv` into that argv. Secret plan entries are skipped when writing (ticket 14's `bootstrap` already wrote them into `target`) — `build_plan` is still called so plain and composed-fragment entries land too, reusing the exact same resolved `decrypted_secrets` dict rather than decrypting twice. Takes an optional `[-- CMD...]` passthrough (mirrors `bin/assemble`'s own convention) purely so `bin/launch-demo` can bound the final hand-off to a quick command instead of the production default (spawn `$SHELL`, which is what a real launch drops into).
- **`flake.nix`** gained `self` in its `outputs` args and one `apps.<system>.default`: a `pkgs.writeShellApplication` (`fzf`, `age`, `sops`, `python3` as `runtimeInputs`, so the launch app carries its own tool deps distinct from the per-bundle packages) whose body is just `exec "${self}/bin/launch"` — `bin/launch` takes no arguments (matches spec: "no arguments, no `#preset` selector"), so nothing is passed through. `${self}` is the flake's own fetched source, so every relative path inside `bin/launch`/`bin/launch-inner` (siblings under the same store path: `presets/`, `bundles/`, `fragments/`, `pkg/`, `identity.age`) resolves correctly under `nix run`.
- **Added a demo-only `identity.age`** at the repo root (age-keygen'd + passphrase-encrypted, passphrase `ephemeral-shell-demo-fixture`, documented in `bin/launch-demo`) — same reasoning as ticket 13's minimal real `vcs`/`personal` pair: `secrets.bootstrap` unconditionally decrypts an identity every launch per the spec ("on every launch..."), so an `identity.age` has to exist for the personal preset (today: `vcs`, no secrets of its own) to be launchable at all. Ticket 16 (real content migration) replaces this with the user's actual identity; nothing here is sensitive.
- **`pkg/dotfiles/testing.py`** is a new tiny shared module (`fake_tty`/`with_passphrase`) extracted from what was originally duplicated pty-driving code in both `tests/test_secrets.py` and `bin/launch-demo` — both need to feed `age`'s interactive passphrase prompt without a real terminal. `tests/test_secrets.py` now imports it instead of defining its own copy (its one non-default call site, encrypting a fresh identity, passes `feed_count=2` for the encrypt/confirm pair; decrypt-only call sites use the new default of 1). Kept out of `pkg/dotfiles/core/` since it isn't resolution logic or a secrets-domain concern, just an OS-facing test/demo helper.
- **`bin/launch-demo`** (Python, for the same pty-driving reason `tests/test_secrets.py` is) checks two things: (1) `build_home` driven against its own throwaway fixture repo (own preset/bundle/fragment tree, `bin`/`pkg` symlinked back to this repo, a plain file + a sops-encrypted secret file + a fragment, all encrypted for the demo identity's own recipient key) — this exercises the actual secret-decrypt and file-write loops (the real `personal`/`vcs` preset has no secrets or files today, so looping over it alone would prove nothing ran); (2) the real `bin/isolate -- bin/launch-inner ... -- true` hand-off, accepting isolate's own documented loud-failure path (no unprivileged userns) as `OK` — this sandbox has the same constraint ticket 12 already hit, so the isolated happy path plus `fzf`'s own interactive UI (needs a real controlling tty, absent in this sandbox — confirmed via bare `tty`) are implemented per spec but unverified live here. Isolation mechanics themselves are already independently proven by `bin/isolate-demo` (ticket 12); package assembly by `bin/assemble-demo` (ticket 13) — this ticket's demo only needed to prove the new glue.
- **Post-implementation `/code-review`** (Standards + Spec axes, run in parallel) caught real gaps, all fixed before commit: (1) `bin/launch-inner` had no argument validation (`IndexError` on bad invocation instead of a usage message) — fixed. (2) `bin/launch-demo`'s pty helpers were a near-verbatim copy of `tests/test_secrets.py`'s — extracted to `pkg/dotfiles/testing.py`, both now import it. (3) `flake.nix`'s wrapper silently swallowed a stray arg via an unused `"$@"` — removed. (4) The real find: the original `bin/launch-demo` only drove the real (secret-less) `personal`/`vcs` preset, so the secrets-decrypt and materialize file-write loops never actually ran even once — only the unconditional identity decrypt was exercised. Rewrote the demo to build its own fixture with a real secret, plain file, and fragment so those loops are genuinely proven.
- **Verified live in this sandbox**: `nix flake show`/`nix flake check` evaluate cleanly (`apps.x86_64-linux.default` present, `shellcheck` clean on `bin/launch` via `writeShellApplication`'s built-in check). `bin/launch-demo` (run under `nix shell nixpkgs#age nixpkgs#sops`) passes both checks, now genuinely exercising secret-decrypt + plain-file + fragment materialization. Full pytest suite (26 tests, unchanged in count — `test_secrets.py` now imports the shared helper instead of defining it) passes via `uv run --with pytest pytest -q`. `ruff check` clean. `nix run . --` piping a preset name at `fzf` fails with `inappropriate ioctl for device` — this sandbox's shell tool has no controlling tty at all (`tty` reports "not a tty"), so the interactive picker itself is implemented per spec but unverified live here; run `nix run .` from a real terminal to confirm.
